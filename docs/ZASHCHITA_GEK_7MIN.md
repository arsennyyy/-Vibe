## Вступление (0:00–1:00)

Уважаемый председатель и члены государственной экзаменационной комиссии!

Тема моего дипломного проекта — **«Веб-платформа +Vibe для продажи билетов и управления мероприятиями»**.

**Актуальность.** Рынок live-событий в Беларуси по-прежнему фрагментирован: афиша на одном ресурсе, схема зала — в кассе, билет — в PDF из почты, возврат — отдельным звонком. Организатор не видит единой картины по продажам и комиссии, а посетитель теряет время на ручной выбор мест и подтверждение оплаты. Цифровая платформа, объединяющая **витрину, бронирование, оплату, билет и контроль на входе**, снижает операционные издержки и повышает прозрачность для всех участников.

**Цель работы** — разработать полнофункциональное веб-приложение, автоматизирующее полный жизненный цикл мероприятия: от создания события организатором до проверки QR-билета на входе.

**Задачи проекта:**
1. Спроектировать клиент-серверную архитектуру и реляционную модель данных (PostgreSQL, 23+ сущности).
2. Реализовать интерактивную **схему зала** с координатной рассадкой реальных минских площадок (8 каталоговых схем, до 4 500 мест).
3. Обеспечить безопасную аутентификацию: **JWT**, двухфакторное подтверждение e-mail по OTP, вход через Google OAuth.
4. Реализовать бизнес-процессы: модерация событий, расчёт **комиссии платформы (12%)**, возвраты, переносы, отмены, **передача билета** между пользователями.
5. Внедрить генерацию **PDF-билетов** и **динамических QR-кодов** с защитой от скриншот-перепродажи.
6. Разработать панели **организатора** и **администратора**, чат поддержки с **ИИ-ассистентом** (Ollama / OpenAI-совместимый API).
7. Провести тестирование (функциональное, API, **E2E Playwright**), экономическое обоснование и раздел по охране труда.

**Технологический стек:** React 18 + TypeScript + Vite (SPA), ASP.NET Core 8 (REST API), Entity Framework Core, PostgreSQL, MailKit, QuestPDF, QRCoder. Интерфейс — адаптивный, тёмная/светлая тема, компонентная библиотека на базе Radix UI.

## Гость — витрина без регистрации (1:00–2:15)

**Страницы:** `/` → `/concerts` → `/event/{id}` → `/faq`

Покажу сценарий **незарегистрированного посетителя**.

На **главной** — поиск по артистам из реального каталога, блок рекомендаций и переход в афишу.

**Каталог концертов** — фильтрация по жанру, дате и типу события; сортировка; карточка «Главное» событие; адаптивная сетка под мобильные устройства.

На **странице мероприятия** — полноэкранная обложка, состав артистов, карта площадки (Яндекс), вкладки «Описание / Схема зала / Отзывы». Ключевая особенность — **интерактивная схема зала на Canvas**: сектора, ряды, VIP и места для маломобильных граждан, подсветка выбранных мест, расчёт суммы в реальном времени, синхронизация занятости с сервером — двойное бронирование исключено.

Доступны юридические страницы, FAQ с админ-редактированием, форма обратной связи с капчей.

## Посетитель — регистрация и покупка (2:15–3:30)

**Страницы:** `/signup` → `/signin` → `/profile` → checkout на `/event/{id}`

Для покупки пользователь проходит **регистрацию с OTP-кодом на e-mail** — это не просто пароль, а подтверждённый канал связи для билетов и возвратов. Альтернатива — **Google Sign-In**.

В **личном кабинете**: аватар с кадрированием, настройки уведомлений, **«Мои билеты»** — каждый билет с **ротирующим QR** (обновление по таймеру), скачивание PDF, запрос возврата, **передача билета** другому пользователю по e-mail с отдельным платёжным сценарием.

Оформление заказа — выбор мест → корзина → **демо-оплата** с фиксацией в таблицах `orders` и `payments`, расщеплением суммы на долю организатора и комиссию платформы. Письмо с PDF уходит через SMTP.

Виджет **чата поддержки**: первичный ответ даёт ИИ по контексту платформы; при необходимости — эскалация живому администратору в панели `/admin`.

## Организатор — конструктор события (3:30–4:45)

**Страницы:** `/profile` → `/organizer/events/{id}/edit`

**Организатор** создаёт событие в пошаговом конструкторе:
- метаданные (название, жанр, дата, площадка из каталога минских залов);
- обложка с **кадрированием под формат hero-баннера**;
- JSON-состав артистов с аватарами;
- привязка **каталоговой схемы зала** и настройка ценовых секторов (`HallThemeJson`);
- отправка на **модерацию**.

После одобрения администратором событие публикуется в каталоге — организатор видит статистику продаж и начислений в разделе «Доходы». Поддерживаются сценарии **переноса даты** и **отмены** с OTP-подтверждением и цепочкой согласования через админ-панель.

## Администратор — управление платформой (4:45–5:45)

**Страницы:** `/admin-signin` → `/admin` → `/admin/preview-event/{id}` → `/verify`

**Администратор** работает в единой панели:
- **дашборд** — выручка, комиссия, количество заказов и пользователей;
- **модерация** событий (одобрение / отклонение с комментарием);
- управление пользователями, назначение роли организатора;
- справочники площадок, FAQ, фильтры каталога;
- обработка обращений, возвратов, запросов на отмену и перенос;
- мониторинг **потоков поддержки** (ИИ + оператор).

Страница **`/verify`** — имитация контроля на входе: сканирование QR, проверка статуса билета, защита от повторного прохода.

---

## Тестирование, экономика, охрана труда (5:45–6:45)

**Тестирование.** Разработан набор **E2E-сценариев Playwright** (публичные страницы, каталог, профиль с моками API). Проведено ручное функциональное тестирование всех ролей, нагрузочная проверка API, валидация целостности БД при каскадных операциях (удаление события, возврат, передача билета).

**Экономическое обоснование.** Рассчитаны трудозатраты разработки, стоимость хостинга и сопровождения. Монетизация — **комиссия 12%** с оборота билетов; в админ-панели отображается раздельный учёт gross / platform fee / organizer payout — это готовая модель для реального внедрения.

**Охрана труда.** В пояснительной записке рассмотрены: эргономика рабочего места программиста, режим труда и отдыха, электробезопасность при работе с ПЭВМ, освещённость и микроклимат.
